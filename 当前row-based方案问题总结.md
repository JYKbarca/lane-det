# 当前 row-based 方案问题总结

## 一、当前方案的阶段性结论

当前方案已经从早期的连续回归版本，演进为统一的网格分类版本，主线逻辑为：

- 固定最多 `K=5` 条车道槽位
- 对每条车道在固定 `Y=56` 个 row 上分别预测 `row valid` 与横向位置
- 图像宽度划分为 `W=100` 个网格
- 车道级输出由 `lane exist`、`row valid`、`row position` 组成
- 仅在有效 row 上做位置分类，并通过期望解码恢复连续坐标

从 `train6.log` 和 `vis_step4` 的结果看，这条技术路径不仅有效，而且前三步改造都已经被训练与可视化验证为有效；第四步也带来了一定的视觉清理收益，但提升已经明显进入边际递减阶段。当前主问题已经从“主车道能否贴住”进一步收敛为“困难场景下为什么仍会残留错误侧枝、局部叉线和异常短段，以及如何在不堆大量后处理的前提下继续提升 hard case 稳定性”。

## 二、目前已经确认的改进

### 1. 表示方式比连续回归更合理

新方案将原来的：

- `exist`
- `valid`
- `x_coords`

改成：

- `lane exist`
- `row valid`
- `row position`

这样做的收益是：

- 将车道是否存在、row 是否有效、横向位置三种语义拆得更清楚
- 分类任务通常比直接回归单个连续坐标更稳定
- 更容易抑制早期出现的“乱飞”“极端折返”“整条假线”问题

### 2. 排序逻辑比旧版本更稳定

`RowTargetBuilder` 已经从“最后 3 个有效点平均 x 排序”改成“所有有效点平均 x 排序”。

这缓解了以下问题：

- 遮挡导致的排序抖动
- 短线样本排序不稳定
- 同一条车道在不同图片中落入不同 slot

### 3. 第一步结构改造已经被训练结果验证为有效

在保持训练配置基本一致的前提下，完成第一步 row head 改造后，训练结果出现了明显跃升：

- 改造前 `train2` 的 best 出现在 `epoch 20`
  - `Accuracy = 0.756733`
  - `FP = 0.578499`
  - `FN = 0.577670`
- 改造后 `train3` 的 best 出现在 `epoch 20`
  - `Accuracy = 0.886498`
  - `FP = 0.183241`
  - `FN = 0.197238`
- 改造后在非常早的阶段就已经明显超过旧版本
  - `epoch 3: Accuracy = 0.822368`
  - `epoch 5: Accuracy = 0.834308`

这说明第一步并不是“结构上更合理但效果未验证”，而是已经实质性解决了旧 head 中“横向空间信息过早丢失”的核心瓶颈。

### 4. 第二步语义拆分已经被训练结果与可视化验证为有效

在完成第二步 `row valid / row position` 语义拆分后，模型效果继续提升：

- `train3` 的 best 出现在 `epoch 20`
  - `Accuracy = 0.886498`
  - `FP = 0.183241`
  - `FN = 0.197238`
- `train4` 的 best 出现在 `epoch 19`
  - `Accuracy = 0.903940`
  - `FP = 0.157643`
  - `FN = 0.173711`

同时从 `vis_step2` 看，step1 中最显眼的“大折线”“大三角形”假线问题已经明显减少，大量直道和缓弯样本中的主车道贴线质量也保持稳定。

这说明第二步并不是单纯的代码重构，而是已经对当前误差模式产生了真实收益，尤其缓解了“少量错误 row 被直接组织成整段假线”的问题。

### 5. 第三步监督与解码对齐已经被训练结果与可视化验证为有效

在完成第三步 soft bin 监督和连续坐标损失之后，模型效果继续提升：

- `train4` 的 best 出现在 `epoch 19`
  - `Accuracy = 0.903940`
  - `FP = 0.157643`
  - `FN = 0.173711`
- `train5` 的 best 出现在 `epoch 25`
  - `Accuracy = 0.904614`
  - `FP = 0.142726`
  - `FN = 0.157413`

同时从 `vis_step3` 看，大量正常样本的主车道贴线精度继续提升，step2 中部分明显的错误折线进一步收缩。

这说明第三步的主要收益是真实存在的，但它更主要体现在：

- 提升贴线精度
- 继续压低 `FP/FN`
- 让错误分支变短、变少

而不是一次性彻底消除所有困难场景下的“飘忽预测线”。

### 6. 第四步轻量连续性改造已经被验证为“有限有效，但提升不再明显”

在完成第四步连续性清理和分段插值之后，`train6` 的 best 出现在 `epoch 20`：

- `Accuracy = 0.906318`
- `FP = 0.159300`
- `FN = 0.172192`

相较 `train5` 的 best：

- `Accuracy` 仅小幅提升
- `FP/FN` 并没有继续下降，反而略有回升

同时从 `vis_step4` 看，部分 hard case 的错误侧枝和大折线确实被缩短，输出更干净了一些；但几个典型顽固样本中的错误分支并没有被根治。

这说明第四步是有效的，但它更像“表达层面的收口”和“可视化伪影的部分清理”，而不是新的主要增益来源。继续在这一层堆更多连续性兜底逻辑，收益很可能已经开始明显变小。

## 三、当前仍然存在的主要问题

## 1. 主车道整体几何已经基本稳定，问题重心继续后移

从 `train6` 和 `vis_step4` 看，当前模型在大量直道和缓弯样本中，主车道已经可以较稳定地贴住 GT，尤其左右边界线和主要中间车道的几何位置相比旧版本、step2 阶段都进一步改善。

这说明：

- 第一阶段和第二阶段最核心的结构瓶颈都已经被明显削弱
- “系统性横向偏移”已经不再是当前主矛盾
- 当前瓶颈已经进一步收敛到“困难场景下的错误分支抑制、局部拓扑稳定性和 hard case 稳定性”

## 2. 当前最突出的残留问题，已经收敛为 hard case 下的错误侧枝、局部叉线和异常短段

从 `vis_step4` 看，虽然第四步已经让一部分错误折线缩短、变干净，但仍会出现少量明显的异常红线，典型形态包括：

- 大三角形或大折返状假线
- 某一段突然横向跳出，再折回主车道
- 遮挡附近或中间车道上出现局部异常段

这说明：

- 当前不是“整条 lane 都预测错了”
- 更像是困难场景下 lane 内部少量 row 或短连续段的判别仍不稳定
- step4 可以减轻其中一部分放大效应，但还没有从源头消掉这些错误分支

## 3. 当前主问题已经不再是单纯连续性表达，而是 hard case 下的判别稳定性仍不够强

前三步已经分别解决了横向空间表达、语义拆分和监督/解码对齐中的主要瓶颈，第四步也对结果表达做了轻量收口。

但从当前残留误差看，新的主要问题是：

- 右侧车道、车辆邻近区域、弯道和并线场景中，模型仍会直接产生错误侧枝或短错误段
- 这些错误并不只是“后续连线太激进”，而是模型在 hard case 上本身就还不够稳
- 因此继续只在解码和可视化层面修补，已经难以带来决定性改善

因此当前最该优先解决的，不再是继续提升纯位置精度，也不是继续堆连续性规则，而是让模型更稳定地学会：

- 困难场景下哪些 row 和短段本来就不应该被预测出来
- 相邻车道、车辆遮挡和弯道结构之间该如何区分
- 哪些局部结构应被保留，哪些局部结构应被自然抑制

## 4. 继续堆第四步这类轻量兜底逻辑，收益已经开始明显变小

`train6 + vis_step4` 已经说明，第四步这类连续性清理是有帮助的，但它带来的提升更偏向“把局部错误画得没那么夸张”，而不是实质性提高困难场景的检测质量。

这意味着：

- 可以保留 step4 现有的轻量约束
- 可以再做一次很小范围的连续性阈值扫描
- 但不应继续沿着“补更多规则、修更多线”的方向深挖

当前更需要把精力转向：

- hard case 数据与增强
- 弯道和特殊场景下的 slot 稳定性
- 更可靠的 checkpoint 选择与固定误差样本分析

## 5. 弯道与特殊场景下的 slot 语义仍需继续关注

排序逻辑改成“所有有效点平均 x”之后，slot 抖动问题已经明显缓解，而且它不再像之前那样是主要矛盾。

但在强弯道和特殊场景下，仍然存在潜在风险：

- 全局平均 `x` 仍可能不能准确代表近场左右关系
- 弯道中不同车道的整体均值可能更接近
- slot 语义不稳时，会进一步放大局部异常点问题

所以当前状态应理解为：

- slot 问题已经退居次要
- 但它仍然可能在困难样本中叠加放大误差

## 6. 现阶段的问题归纳

如果只保留现阶段最需要解决的几个问题，可以归纳为：

- 主车道已经大体贴住，但困难场景下仍会残留少量错误侧枝、局部叉线或异常短段
- 残留问题主要集中在右侧车道、车辆邻近区域、弯道和并线场景
- 第四步已经证明“轻量连续性清理”只能部分缓解问题，继续深挖这条线的收益会越来越小
- 当前更像是“hard case 判别稳定性和泛化不够”，而不是“整条车道整体学不会”
- 不应再把重点放在堆兜底后处理上，而应转向更针对 hard case 的训练、排序和分析流程优化

## 7. 当前 best checkpoint 选择标准不够全面

目前 `best.pth` 仍然只按 `Accuracy` 选择。

这存在问题：

- `Accuracy` 高不一定代表 `FP/FN` 更平衡
- 当前阶段视觉效果和评估数值并不完全同步
- 单独按 `Accuracy` 选 best，可能错过更平衡的 checkpoint
- `train6` 的 `Accuracy` 略高于 `train5`，但 `FP/FN` 反而更差，就是一个直接例子

因此当前 checkpoint 选择标准存在明显优化空间。

## 四、已经排除或弱化的问题

## 1. 不是 Backbone/FPN 失效

当前 `ResNet18 + FPN` 明显在正常起作用：

- loss 在正常下降
- 模型能输出接近车道的结构
- 指标在完整训练过程中持续改善，并且在第一步结构改造后出现了明显跃升

所以当前的主问题不在 backbone，而在：

- 输出表示
- slot 对齐
- 几何监督与解码方式

## 2. 不是简单阈值问题

之前已经做过多组阈值扫描，视觉主形态没有本质变化。

说明：

- 不是单纯 `exist_score_thr` 太低
- 也不是后处理阈值过松导致的假线泛滥

问题更深层，主要在模型本身输出的 lane 语义和几何质量上。

## 3. 不是继续调平滑权重就能解决的问题

之前尝试过更强的平滑损失和 query 正则，结果表明：

- 线条会更顺
- 但会更模板化
- 同时更容易整体偏移

说明单纯加强几何约束不是主线解决方案。

## 五、当前方案相比旧方案的真实位置

当前 grid-based 方案相较旧的连续回归版，处于如下状态：

- 在技术方向上更合理
- 在实验结果上已经形成明显领先
- 前三步结构与监督改造都已经被验证为有效，第四步也有一定收益但边际提升已变小
- 当前主瓶颈已经从“整体贴线能力不足”转向“hard case 下的错误分支抑制、泛化与稳定性”

也就是说：

它已经不是“试错失败的分支”，而是“值得继续作为主线推进的方案”。

## 六、当前最核心的问题归纳

如果只保留一句总结，当前问题可以概括为：

> 新方案已经学会了在大多数样本上更稳定地贴住主车道，但仍然没有稳定学会“在右侧车道、车辆邻近、弯道和并线这些困难场景下，哪些局部结构根本不应该被预测出来，以及如何避免少量错误分支残留到最终结果里”。

进一步拆开就是三点：

1. 主体结构已经大体对上，但 hard case 下仍会残留错误侧枝和局部叉线
2. 第四步已经验证“只做连续性清理”不能继续带来大幅提升
3. slot 对齐和 hard case 泛化仍会在弯道、遮挡和并线场景里叠加放大误差

## 七、后续工作建议

后续不建议再回到旧的连续回归框架，而应继续沿当前 grid-based 主线推进。

但接下来的推进方式需要更明确地按主线步骤展开，避免“哪里有问题就加一点兜底逻辑”的碎片化优化。

### 0. 实现原则

后续所有优化都应遵循下面的原则：

- 优先修改真正决定检测效果的主干环节：表示、监督、解码、训练数据
- 优先做能提升“贴线精度、连续性、困难场景鲁棒性”的主线改动
- 尽量避免堆积大量兜底规则、特判后处理和临时修补逻辑
- 不要为了让个别样本看起来更顺而明显牺牲代码可读性、结构清晰度和整体可维护性
- 如果某个改动主要收益来自“补洞式修线”而不是“让模型本身学得更准”，应降低优先级

### 当前优先级调整（基于 `train6 + vis_step4`）

1. 不再继续扩展 step4 这类连续性兜底逻辑，只保留现有实现，最多做一次很小范围的参数扫描
2. 最高优先级转向困难场景导向的数据与增强，让模型本身更少产生错误侧枝和异常短段
3. 继续优化弯道和特殊场景下的 slot 排序稳定性，减少结构性错位
4. 同步重做 checkpoint 选择与固定 hard case 分析流程，避免被单一 `Accuracy` 指标误导

### 下一步具体方案：先做 hard case 导向的数据与分析闭环

这一阶段不建议再继续改 head、loss 或堆更多修线规则，而是优先做一轮“小而准”的 hard case 数据与增强改造。具体按下面顺序推进：

1. 先固定一组 hard case 对照样本
2. 再补一批只服务当前问题的数据增强
3. 最后用统一标准比较 baseline 和新实验

这样做的核心意图不是“把增强做多”，而是让后续每一次训练都能准确回答：模型是否真的更少地产生错误侧枝和异常短段。

具体方案如下：

#### A. 固定 hard case 对照集

先从当前最典型的问题样本中固定一小组对照图，建议规模为 `20~40` 张，优先覆盖：

- 右侧车道旁边有车辆遮挡的样本
- 弯道和并线场景
- 中间车道容易长出错误侧枝的样本
- 之前在 `vis_step3 / vis_step4` 中反复出现问题的顽固样本

这一步意图解决的点是：

- 避免只看整体 `Accuracy`，却忽略 hard case 是否真的变好
- 让后续增强和排序改动有稳定、可复现的对照基准
- 防止“某次训练数值略升，但顽固假线其实没变”这种误判

#### B. 第一批增强：先做不改标注的 image-only hard case 增强

优先加入三类不需要改车道标注的增强：

1. `RandomOcclusion / Cutout`
2. `RandomShadow / Brightness Drop`
3. `RandomBlur`

建议控制在轻量范围内：

- 遮挡块数量少、面积中等，只模拟邻车、阴影或局部遮挡
- 阴影和亮度变化只做中等幅度，不做极端黑图
- 模糊只做轻度，避免把整张图破坏成不真实样本

这一步意图解决的点是：

- 右侧车辆邻近区域容易长出错误侧枝
- 遮挡附近 row 判别不稳
- 光照变化和局部模糊导致的碎段和异常短段

之所以先做这一步，是因为它实现简单、风险低，而且最贴近当前暴露出来的误差来源。

#### C. 第二批增强：补小幅几何扰动，但只做真实且可控的扰动

在 image-only 增强稳定之后，再加一批轻量几何增强，优先考虑：

1. 小幅横向平移
2. 小幅纵向平移
3. 小幅透视/仿射扰动

这里要严格限制幅度，不做夸张几何变形。原则是：

- 只模拟真实摄像头姿态和车身抖动可能带来的视角变化
- 扰动后仍应保持车道结构自然
- 标注映射必须准确，不能为了增强而引入脏监督

这一步意图解决的点是：

- 弯道和并线场景中的结构不稳
- 近场与远场透视变化导致的局部错误分支
- slot 排序和局部几何关系在轻微视角变化下不够稳的问题

#### D. 实验顺序必须做成小步 ablation

不建议一次把所有增强全加上去，而应按下面顺序做：

1. baseline：当前 step4 版本
2. aug-v1：只加 image-only hard case 增强
3. aug-v2：在 aug-v1 基础上再加轻量几何增强

每一步都只改一个方向，这样才能明确判断收益来自哪里。

这一步意图解决的点是：

- 避免增强项互相叠加后无法判断谁真正有效
- 避免“配置变复杂了，但不知道哪一项在起作用”
- 保持代码和实验结论都清晰可解释

#### E. 评估标准要从单一 Accuracy 改成三层判断

每次实验都同时看三层结果：

1. 全量验证集 `Accuracy / FP / FN`
2. 固定 hard case 对照集的可视化变化
3. 顽固错误样本里错误侧枝、局部叉线和异常短段是否明显减少

如果出现下面这种情况，就不应视为真正进步：

- `Accuracy` 略涨
- 但 `FP/FN` 变差
- hard case 图里老问题还在

这一步意图解决的点是：

- 避免被单一指标误导
- 让后续“best checkpoint”定义更接近真实检测质量
- 确保优化方向真正服务当前主问题，而不是服务某个单独数值

### 1. 第一步：优先修正 row head，使其保留横向空间信息（已完成，并被 `train3 + vis_step1` 验证有效）

当前最优先的问题，不是继续调阈值或加后处理，而是 head 本身对横向位置信息利用不够充分。

当前已完成的改动：

1. 不再过早把宽度维压成 `1`
2. 先保留 `row-grid` 横向结构，再进行位置分类
3. 将每个 lane slot 的位置打分建立在显式的 row-grid 特征上，而不是仅依赖全局压缩后的 row 特征
4. 将“no-lane”打分从位置网格打分中分离出来，保持语义更清晰

本次完成的是结构层面的第一步改造，而且已经被后续训练和可视化验证为有效。

这一改动直接对应的核心意图是：

- 不让模型在预测 `x` 网格之前先丢掉横向空间信息
- 让中间车道和相邻车道之间的横向差异能够更直接地被建模
- 从表示层面减轻“模板线”和整体横向偏移问题

本步骤已经确认改善了：

- 整体横向偏移
- 中间车道定位质量
- 弯道场景下的几何贴合能力

### 2. 第二步：优先拆清 row 是否存在车道 与 row 的横向位置 两种语义（已完成，并被 `train4 + vis_step2` 验证有效）

当前 `W+1` 分类把“位置”和“该 row 无车道”混在一起，这已经成为现阶段最优先的新瓶颈。

当前已完成的改动：

1. 将输出拆成 `lane exist`、`row valid`、`row position` 三部分
2. 不再让位置分类承担“该 row 无车道”的混合语义
3. 只在有效 row 上计算位置分类损失
4. 解码时先根据 `row valid` 判断哪些 row 应该保留，再恢复横向坐标
5. 将无效 row 的 `grid target` 改为忽略索引，避免无效点参与位置分类训练

本步骤完成的是语义拆分和训练/解码链路改造，当前已经被后续训练与可视化验证为有效。

已经确认的收益是：

- 指标继续提升，`Accuracy` 首次稳定进入 `0.90` 区间
- step1 中最显眼的大折线假线频率明显下降
- 主车道贴线质量得以保留，同时异常 row 被放大成整段假线的问题得到缓解

但这一步并没有彻底消除困难场景下的局部异常段，所以它解决的是主问题的一大块，而不是全部。

这一阶段的目标是：

- 减少局部异常点
- 减少大折线、大三角形假线
- 提高遮挡和中间车道上的 row-to-row 稳定性

### 3. 第三步：对齐监督与解码，重点提升贴线精度（已完成，并被 `train5 + vis_step3` 验证有效）

在 step1 和 step2 都已被验证有效之后，当前最应该推进的主线就是监督与解码对齐。当前训练时是离散分类监督，推理时再通过期望解码恢复连续坐标，这两者之间还不够对齐。

当前已完成的改动：

1. 在现有 grid 分类之外，增加了直接约束 `expected_x` 的连续坐标损失
2. 将原先依赖单个 hard bin 的位置监督改成了更接近期望解码语义的 soft bin 监督
3. 训练目标现在会直接鼓励 `expected_x` 更贴近真实车道位置，而不只是命中某一个离散网格
4. 保留了现有主干结构，没有额外引入复杂兜底后处理

本步骤当前已经被后续训练与可视化验证为有效。

已经确认的收益是：

- 主车道贴线精度继续提升
- `FP/FN` 继续下降
- 部分错误折线明显缩短或收缩

但这一步并没有彻底解决困难场景下的错误侧枝和局部拓扑异常，所以它解决的是“位置更准”，不是“连续段一定更干净”。

这一阶段的目标是：

- 减少整条线系统性偏内/偏外
- 提升亚网格级别的位置精度
- 让可视化结果从“接近 GT”进一步变成“贴住 GT”

### 4. 第四步：以轻量方式加强局部连续性，并让转换表达尊重有效连续段（已完成，并被 `train6 + vis_step4` 验证为有限有效）

当前局部锯齿、轻微折返和前后段衔接不自然，说明连续性约束还不够，但这不意味着应该堆大量后处理规则。

当前已完成的改动：

1. 在解码阶段增加了基于连续段长度和局部横向跳变的轻量清理
2. 仅保留满足最小连续段要求的有效段，避免短异常段直接保留下来
3. 在 TuSimple 转换阶段按连续有效段分段插值，不再跨明显断点强行连线
4. 在可视化阶段尊重 `-2` 缺口，不再把断开的预测段画成连续折线

本步骤当前已经被后续训练与可视化验证为“有限有效”：

- `train6` 的 `Accuracy` 略高于 `train5`
- 但 `FP/FN` 没有继续下降，说明它不是全指标一致变好
- `vis_step4` 中部分错误侧枝和大折线变短、变干净
- 但顽固 hard case 并没有被根治

这一结果说明：

- step4 是值得保留的轻量收口
- 但它不是下一阶段继续深挖的主增益来源
- 后处理只应保留最基础、最稳定的连续性约束，不应继续膨胀成“第二个模型”

### 5. 第五步：继续优化弯道和特殊场景下的 slot 排序稳定性

“所有有效点平均 x 排序”已经比旧版本更稳，但它仍然更像过渡方案，而不是最终方案。

需要做的事情：

1. 重点分析弯道场景下 slot 是否仍存在错位
2. 让排序依据更贴近近场真实左右关系，而不是只依赖全局均值
3. 避免因为排序不稳放大几何偏移

这一阶段的目标是：

- 减少弯道样本中的 slot 语义漂移
- 减少中间车道在特殊场景中的错分和偏移

### 6. 第六步：补强训练数据和增强方式，但只做真正服务主问题的增强（当前最高优先级，代码改造已完成，效果待训练验证）

当前数据增强主要是翻转和颜色扰动，对弯道、遮挡、局部断裂这类问题帮助有限。

当前已完成的代码改动：

1. 已加入 `RandomOcclusion / Cutout`
2. 已加入 `RandomShadow / Brightness Drop`
3. 已加入 `RandomBlur`
4. 已加入轻量 `RandomTranslateShear`，用于后续小幅几何增强 ablation
5. 当前默认配置先启用 image-only hard case 增强，几何增强能力已接好但默认未打开，便于按 `baseline -> aug-v1 -> aug-v2` 顺序推进

需要做的事情：

1. 先补 `RandomOcclusion / Cutout`、`RandomShadow / Brightness Drop`、`RandomBlur` 这类 image-only hard case 增强
2. 再补小幅横向/纵向平移和小幅透视/仿射扰动，重点覆盖弯道、并线和视角轻微变化
3. 所有增强都控制在真实、轻量、可解释的范围内，不做夸张扰动
4. 必须按 baseline -> aug-v1 -> aug-v2 的顺序做小步 ablation
5. 每轮都结合固定 hard case 对照集判断是否真的减少错误侧枝和异常短段

这一阶段的目标是：

- 提升右侧车道、车辆邻近、弯道和并线场景下的判别稳定性
- 让模型源头上更少产生错误侧枝、局部叉线和异常短段
- 提升困难场景鲁棒性和泛化质量
- 让改进直接服务当前暴露出来的误差模式，而不是只改善普通样本

### 7. 第七步：重做 checkpoint 选择与误差分析流程（与第六步同步推进）

当前只按 `Accuracy` 选 best 不够全面，也会影响对模型真实水平的判断。

需要做的事情：

1. 联合考虑 `Accuracy`、`FP`、`FN` 和 hard case 可视化质量
2. 固定一套统一的 best 模型筛选标准
3. 固定一组 hard case 对照样本，每轮都看同一批图
4. 每轮重点看中间车道、遮挡区域、弯道样本和“异常折线假线”误差模式

这一阶段的目标是：

- 让“最佳模型”定义更可靠
- 避免被单一指标误导
- 让后续优化建立在更可信的分析基础上
- 让 hard case 上的真实进步能够被稳定识别出来

### 8. 第八步：在主结构改完后，再确认训练上限

从 `train6` 看，模型在 `epoch 20` 达到当前按 `Accuracy` 选择的 best，后续并未继续明显提升，说明当前已经进入高位平台区间；在 hard case 主问题尚未被解决前，单纯继续拉长训练仍不是最高优先级。

需要做的事情：

1. 先完成前面的关键结构改动
2. 再重新观察学习率调度和训练轮数的上限
3. 在新结构上确认是否还需要更长训练

这一阶段的目标是：

- 避免在次优结构上反复堆训练时间
- 把训练资源优先用在正确的模型形态上

## 八、当前结论

当前 row-based 技术路径已经完成了一次重要升级：

- 从连续回归转向统一网格分类
- 从分裂的 `valid + coord` 表示转向统一的结构化位置分类

它已经证明自己比旧版本更有潜力，而且当前训练过程也显示出明确的持续收敛趋势；但它还没有完成最终收敛。

现阶段最准确的判断是：

> 这条路径值得继续，而且前三步关键改造都已经被稳定验证有效，第四步也证明了轻量连续性收口有一定价值；但当前提升已经开始进入平台期，接下来真正该解决的不是“再怎么修线”，而是“如何让模型在右侧车道、车辆邻近、弯道和并线这些 hard case 中更少地产生错误分支，并用更可靠的评估与选模流程准确识别真实进步”。

### 排序实现补充（已完成）
`RowTargetBuilder` 当前已经不再使用“所有有效点平均 x 排序”作为最终实现，而是改成了：

1. 优先使用底部最近 `6` 个有效点的加权平均 `x`
2. 越靠近底部的点权重越大
3. 如果有效点少于 `3` 个，则回退到所有有效点平均 `x`

这样做的意图是：

- 比旧的“最后 3 个有效点平均 x”更稳，不容易因为遮挡和短线剧烈抖动
- 比“所有有效点平均 x”更贴近近场真实左右关系
- 减少弯道、并线和邻近车道干扰下的 slot 结构性错位

当前验证流程已经补充了“完整性诊断”指标，用来区分当前的短线问题到底更像：

1. 模型在困难区域本身就偏保守，不愿意继续给出有效 row
2. 还是连续性清理和后续解码把原本存在的有效 row 又删掉了

新增指标包括：

- `RawRowRecall`：阈值化后的 `row_valid` 在 GT 有效 row 上的召回
- `FinalRowRecall`：连续性清理后的最终有效 row 在 GT 有效 row 上的召回
- `UpperRowRecall`：图像上半部分 GT 有效 row 的最终召回
- `LowerRowRecall`：图像下半部分 GT 有效 row 的最终召回
- `CleanupKeepRate`：连续性清理后保留下来的 GT 命中占原始命中的比例

这一组指标的意图是：

- 不再只靠 `Accuracy / FP / FN` 猜测“线为什么变短”
- 直接判断短线主要出在 `row_valid` 召回不足，还是出在 continuity 清理过强
- 直接判断问题更集中在上半部、中上部，还是整条线整体都在丢失

如果后续出现下面两种模式，就可以直接定位方向：

- `RawRowRecall` 就明显偏低：说明主问题更偏训练侧，模型本身在 hard case 里变保守了
- `RawRowRecall` 还可以，但 `FinalRowRecall / CleanupKeepRate` 明显掉下去：说明主问题更偏解码和连续性清理
